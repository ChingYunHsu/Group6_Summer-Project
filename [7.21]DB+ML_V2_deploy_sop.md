# ClearPath V2 deployment SOP

Last updated: 2026-07-18

## Fresh DB + ML deployment

Use this section for a new machine or a deliberately rebuilt local database.
It is destructive: `docker compose down -v` removes the local MySQL and Redis
volumes. Do **not** run it when the existing database must be retained.

Before starting, make sure these deployment inputs are available. The venue
snapshot and V2 label artifacts below are versioned with the repository; the
raw source bundle remains a local-only dependency:

- `venues_export.sql` at the repository root (the current, Git-tracked
  venue-data snapshot)
  plus its [31-column schema contract](docs/venues_export_20260718_schema.md);
- `../data_source/` (the raw source bundle, required only to rebuild omitted
  accessibility data from source);
- `Data+ML/test/7.13-7.18/output/serpapi_v2_labels_20260716/` and
  `Data+ML/test/7.13-7.18/output/serpapi_repeat_audit/` (V2 label artifacts).

### 1. Create schema and restore venue data

```bash
cd /path/to/Group6_Summer-Project

# Fresh database only: removes existing local data.
docker compose down -v
docker compose up -d mysql redis
docker compose exec -T mysql mysqladmin ping -h localhost -uclearpath_app -pclearpath_app

# Required for an already-created DB that predates the V2 label fields.
docker exec -i clearpath-mysql mysql -uclearpath_app -pclearpath_app clearpath \
  < docker/mysql/init/012_add_venues_prediction_label_columns.sql

# Docker init scripts create the schema and five smoke-test seed venues.
# venues_export.sql is a legacy data-only dump with positional VALUES.
# First convert its `venues` insert to a column-named import using
# docs/venues_export_20260718_schema.md; do not import the raw file into an
# unknown existing schema.
docker exec -i clearpath-mysql mysql -uclearpath_app -pclearpath_app clearpath \
  < venues_export_converted.sql

# These migrations must run after the snapshot: they classify medical subtypes
# and mark healthcare records without verified accessibility data as `unknown`.
docker exec -i clearpath-mysql mysql -uclearpath_app -pclearpath_app clearpath \
  < docker/mysql/init/010_backfill_healthcare_venue_subtypes.sql
docker exec -i clearpath-mysql mysql -uclearpath_app -pclearpath_app clearpath \
  < docker/mysql/init/011_backfill_unverified_healthcare_metadata.sql
```

`venues_export.sql` does not include `venue_accessibility`. To restore the OSM
wheelchair annotations as well, run the source ETL notebook after the snapshot
restore, with `CLEARPATH_DATA_ROOT` pointing at the copied source bundle. Its
healthcare ETL is idempotent and upserts the accessibility rows:

```bash
cd /path/to/Group6_Summer-Project
export CLEARPATH_DATA_ROOT="$(cd ../data_source && pwd)"
source .venv-1/bin/activate
jupyter notebook Data+ML/test/6.2-6.5_DB/database_build.ipynb
```

Run the notebook's source-load and four ETL cells (`restrooms`, `AED`,
`healthcare`, `ramps`) in order. This step is required only when the source
bundle is available and OSM accessibility coverage must be restored; otherwise
the API accurately reports the affected venues as `unknown`.

### 2. Publish DB + ML artifacts

Run the static traffic baseline and then publish the V2 forecast curve:

```bash
cd /path/to/Group6_Summer-Project

.venv-1/bin/python Data+ML/test/6.15-6.20/src/busyness_ingestion.py --year 2025

cd Data+ML/test/7.13-7.18
../../../.venv-1/bin/python forecast_v2_pattern.py \
  --labels output/serpapi_v2_labels_20260716/serpapi_popular_times_weak_labels.csv \
  --legacy-labels output/serpapi_repeat_audit/legacy_cached_baseline.csv \
  --output-dir output/v2_pattern_traffic_latest \
  --publish
```

The traffic script writes the retained `nyc_traffic_baseline_v1` hour-of-day
pattern and retires obsolete `nyc_traffic_context_v1` rows. It does not create
a false wall-clock traffic window. The V2 script writes
`model_version='forecast-v2'` into `busyness_forecasts`.

### 3. Start API and web client

Copy `backend/.env.example` to `backend/.env` and set at least `API_KEY`, DB
credentials, `JWT_SECRET`, and `MEDICAL_PROFILE_ENCRYPTION_KEY`. Then use two
terminals:

```bash
# Terminal A: Flask API
cd /path/to/Group6_Summer-Project/backend
../.venv-1/bin/python src/app.py
```

```bash
# Terminal B: React/Vite client
cd /path/to/Group6_Summer-Project/frontend/web
npm install                    # first run or package-lock change only
export VITE_API_KEY='same value as backend API_KEY'
npm run dev
```

Open `http://localhost:5173/`. Vite proxies `/api` to `127.0.0.1:5000`.

## Patch deployment

From the project root, start MySQL and Redis. Set production values for
`API_KEY`, database credentials, and external-service keys before starting the
API.

```bash
docker compose up -d mysql redis

cd backend
python src/app.py
```

In a second terminal, refresh the retained NYC SODA hour-of-day baseline and
publish V2's 12-hour curve. `--year 2025` identifies the source-data year; it
is not a current-status window.

```bash
cd /path/to/Group6_Summer-Project

.venv-1/bin/python Data+ML/test/6.15-6.20/src/busyness_ingestion.py \
  --year 2025

cd Data+ML/test/7.13-7.18
../../../.venv-1/bin/python forecast_v2_pattern.py \
  --labels output/serpapi_v2_labels_20260716/serpapi_popular_times_weak_labels.csv \
  --legacy-labels output/serpapi_repeat_audit/legacy_cached_baseline.csv \
  --output-dir output/v2_pattern_traffic_latest \
  --publish
```

The traffic script writes the retained `nyc_traffic_baseline_v1` hour-of-day
pattern for V2 and removes obsolete `nyc_traffic_context_v1` rows.  Real-time

The V2 script upserts its output as `model_version='forecast-v2'` in
`busyness_forecasts`.

Deploy the mobile client with its API base URL pointing at this API. The map's
initial request is `GET /api/v1/venues` with no filters.

## Basic post-deploy check

Run this against the deployed API.  It fails if the endpoint is unavailable,
if its `count` does not match the returned `items`, or if it returns five or
fewer venues.  This catches the DB-to-mock fallback and accidental list
truncation that would make the frontend show only five venues.

```bash
export CLEARPATH_API_URL='http://127.0.0.1:5000'
export API_KEY='your-deployed-api-key'

curl -fsS \
  -H "X-API-Key: $API_KEY" \
  "$CLEARPATH_API_URL/api/v1/venues" \
  | python3 -c 'import json, sys; payload = json.load(sys.stdin); count = payload.get("count"); items = payload.get("items", []); assert isinstance(count, int), "missing integer count"; assert count == len(items), f"count={count}, items={len(items)}"; assert count > 5, f"only {count} venues returned"; print(f"PASS: {count} venues returned")'
```

Then open the mobile map with no filters.  Its marker count must match the
printed API count.  If it does not, check the mobile client API base URL and
the active filters before changing the model or database.

## Busyness checks

Use a known V2 venue from the just-written curve (the second CSV line contains
one) and confirm both the current busyness source and the 12-hour V2 forecast.

```bash
cd /path/to/Group6_Summer-Project/Data+ML/test/7.13-7.18
VENUE_ID=$(sed -n '2p' output/v2_pattern_traffic_latest/prediction_curve_v2_pattern.csv | cut -d, -f1)

curl -fsS -H "X-API-Key: $API_KEY" \
  "$CLEARPATH_API_URL/api/v1/venues/$VENUE_ID/busyness" \
  | python3 -c 'import json,sys; p=json.load(sys.stdin); assert p["busyness"]["data_mode"] in {"live", "baseline"}; print("PASS: current busyness source")'

curl -fsS -H "X-API-Key: $API_KEY" \
  "$CLEARPATH_API_URL/api/v1/venues/$VENUE_ID/busyness/forecast" \
  | python3 -c 'import json,sys; p=json.load(sys.stdin); assert p["forecast_source"] == "busyness_forecasts"; assert p["model_version"] == "forecast-v2"; assert len(p["forecast"]) == 12; print("PASS: V2 12-hour forecast")'
```
